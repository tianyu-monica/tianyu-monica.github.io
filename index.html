L7 Interprocedural Analysis

Posted on 2021-10-27 In Static Analysis
Motivation

使数据流信息在程序间控制流边上传播，可得到更好的精度。而不必对方法调用采取最保守的假设。

Call Graph Construction(CHA)

**调用图(Call Graph)**是程序中调用关系的表示，是一组从调用点指向目标方法的调用边的集合。

针对面向对象语言的调用图构建：

  • Class hierarchy analysis (CHA)
  • Rapid type analysis (RTA)
  • Variable type analysis (VTA)
  • Pointer analysis (k-CFA)

从上到下越来越准确，从下到上越来越高效。

java中的方法调用

静态调用：调用静态方法，只有一个目标方法，静态分析时可判断出调用对象

特殊调用：调用构造函数、私有实例方法、父类实例方法，只有一个目标方法，静态分析时可判断出调用对象

虚拟调用：其他实例方法的调用，为实现多态，其目标方法可能存在多个，运行时才能最终确定。（构建调用图的关键）

虚拟调用的方法调度

运行时，一个虚拟调用最终如何解析，取决于：

  • 方法接收对象的类型
  • 调用点处的方法签名（此处指：class type+method name+descriptor（return type+parameter types）用以标识一个方法）

记对象类型为$c$，方法签名为$m$，定义调度函数$Dispatch(c,m)$，用以模拟运行时的方法调度过程。

接收对象所属类中找不到对应的非抽象方法时，向父类寻找。

Class Hierarchy Analysis

需要知道整个程序的类继承信息。基于接收对象的声明类型（精度不高的原因）对虚拟调用进行解析。即，假设该对象可能指向此类型及此类型所有子类，通过搜寻它的类继承完成目标方法的解析。

CHA中，定义解析函数$Resolve(cs)$，用以解析一个调用点处所有可能的目标方法。只有虚拟调用存在不确定性，需遍历整个继承树中的类。

Features of CHA

  • fast

    • 只需考虑调用点处的接收对象的声明类型及其继承类
    • 忽略了数据流、控制流信息

  • imprecise

    • 易导致不正确的目标方法

普遍应用：IDE

Call Graph Construction

通过CHA为整个程序构建调用图

  • 从Entry方法开始
  • 对每一个可达的方法$m$，通过CHA，对其中每一个调用点进行目标方法的解析
  • 重复，直到没有新的方法被发现

Interprocedural Control-Flow Graph

控制流图表示独立方法内部的结构，程序间控制流图 (ICFG) 可表示整个程序的结构，是程序间分析的基础。

ICFG包含了每个方法的CFG，并增加了两种类型的边：

  • 调用边：从调用点指向调用目标的入口结点
  • 返回边：从调用目标的出口结点指向紧跟在调用点后的语句

即：$ICFG=CFGs+call\ &\ return\ edges$

而调用图提供了连接这两种边的信息。

  • call-to-return edges存在的意义？

    propagate local data-flow

Interprocedural Data-Flow Analysis

基于ICFG，可实现程序间数据流分析。就迁移函数而言，其相较于CFG的$node\ transfer$，增加了$edge\ transfer$。

  • Call edge transfer: transfer data flow from call site to the entry node of callee
  • Return edge transfer: transfer data flow from exit node of the callee to the return site
Interprocedural Constant Propagation

  • Call edge transfer: pass argument values

  • Return edge transfer: pass return values

  • Node transfer: same as intraprocedural constant propagation except that for call nodes, the transfer function is identity function.

结点迁移函数中，对于调用结点不作处理，而是将调用方法的返回值留给边迁移函数处理。但需在call-to-return边中，kill LHS 变量。

基于格的数据流分析框架

数据流分析可以看作: 对格中元素迭代地应用迁移函数与合并操作

单调性和不动点定理

迭代算法与不动点定理关联

k个结点对应乘积格$L^k$ ，且当$L$为全格时，$L^k$也为全格

每一次迭代，等价于: 应用函数$F$​，其包含了迁移函数与合并操作

其中，迁移函数是单调的，经证明合并操作亦单调. 即: 不动点定理可用于迭代算法.

何时到达不动点

格的高度$h$为从顶端到底端的最长路径的长度. 记CFG有$k$个结点，若每次迭代中，仅有一个结点的值在格上前进一步，则最多需要$h\times k$次迭代.

格视角下的may/must分析

以reaching definition analysis的may analysis为例.

既然是may，那么“所有定义都具有可能性”是完全正确的，只是没有作用. 因此Top是最安全的一端，与此相对，Bottom是unsafe的一端. 同时，对于每个分析对象，必然有一个客观存在的真相，只是算法未必能精确找. 但它的存在将格中元素划分为safe与unsafe两部分，over-approximation就是尽可能向safe端over，以保证正确性.

在over-approximation下，达到的第一个不动点便是最佳解，因为函数的单调性使得越往后执行，距离格的顶端越近，即: 离真相越远，精度下降.

MOP和分配性
Meet-Over-All-Paths Solution(MOP)

MOP分别计算每条路径所得value，并在末尾处将其合并，取得最小上界或最大下界. 但由于有些路径在运行时可能不会被执行，所以MOP并不完全精确. 此外，实际程序中往往有数以亿计的路径，路径上更有很多结点，该方法并不实际可行. 但它提供了一个精度的衡量.

证明可得: MOP更加接近truth，更为精准.

当$F$​满足分配性时，两者相等. (gen/kill problems 都可分配)

L3-4 Data Flow Analysis (Applications)

Posted on 2021-10-27 In Static Analysis
数据流分析总览

静态分析关键词: Abstraction，Over-approximation

may analysis:

Outputs information that may be true(over-approximation)

must analysis:

Outputs information that must be true(under-approximation)

over- and under-approximation are both for safety of analysis

准备知识
Input and Output States

  • each execution of an IR statement transforms an input state to a new output state

  • the input (output) state is associated with the program point before (after) the statement

数据流分析的目的是: 为所有语句的输入输出找到一组，基于控制流和迁移函数的，安全近似约束的解决方案.

基于迁移函数的约束

Forward Analysis: $OUT[s]=f_s(IN[s])$

Backward Analysis: $IN[s]=f_s(OUT[s])$

基于控制流的约束

Control flow within a BB: $IN[s_{i+1}]=OUT[s_i],for\ i=1,2,……,n-1$​

Control flow among BBs: $IN[B]=IN[s_1],OUT[B]=OUT[s_n]$

可达定义分析(Reaching Definitions Analysis)

A definition d at program point p reaches a point q if there is a path from p to q such that d is not killed along that path.

即: 在p点对变量v进行赋值，从p到q的路径中，没有对v的重新定义

可用来检测可能的未定义变量：在entry中引入虚拟定义，若能reach某个点，则该变量可能未定义.

  • Abstraction

    Data Flow Values: the definitions of all the variables in a program

  • Safe-approximation

    Transfer Function: $OUT[B]=gen_B\cup(IN[B]-kill_B)$​

    某语句中定义了变量v，即为generate了该定义，并将其他语句中v的赋值kill

    Control Flow: $IN[B]=\cup_{P\ a\ predecessor\ of\ B}OUT[P]$​

    (只要有一条路径可使v到达某点，则v可达该点. 因此取并集)

Algorithm of Reaching Definitions Analysis

第一行单独初始化boundary: 作为一个经典迭代算法的模板，可适用所有数据流分析，存在其他分析方法使boundary和其他结点初始化值不一样的情况.

算法为什么能停？

对于每个BB，$gen_{B}$和$kill_B$在每次迭代过程中保持不变，仅当$IN[B]$改变时，$OUT[B]$​有可能改变，对于每个加入$OUT[B]$的fact，或者在$gen_B$中，或者未被kill，无论哪种情况，其都会一直留存. 即: $OUT[B]$​单调增加. 同时，fact的集合有限，因此必能停止.

当满足条件: 任何$OUT$都不再改变时，循环停止. 显然，$IN$作为$OUT$的并集，也随之不再改变. 同时，$gen_B,kill_B$​本身不变，算法达到一个不动点(fixed point).

存活变量分析(Live Variables Analysis)

Live variables analysis tells whether the value of variable v at program point p could be used along some path in CFG starting at p. If so, v is live at p; otherwise, v is dead at p.

可用于寄存器分配: 无空寄存器时，使用寄存器优先考虑存储值dead的寄存器.

  • Abstraction

    Data Flow Values: all the variables in a program

  • Safe-approximation

    适用后向分析: 从后往前搜索，只要在某点找到一个变量v的使用，即证明在此之前可达该点的，定义了v的程序点处，v都是live.

    Transfer Function: $IN[B]=use_B\cup(OUT[B]-def_B)$​

    同一变量，先use，再redefine，仍然存活. 未使用就重新定义则dead.

    Control Flow: $OUT[B]=\cup_{S\ a\ successor\ of\ B}IN[S]$

    只要变量v之后有一条路径可使用，则该变量为live，因此使用并集

Algorithm of Live Variables Analysis

可用表达式分析(Available Expression Analysis)

An expression x op y is available at program point p if:

(1)all paths from the entry to p must pass through the evaluation x op y

(2)after the last evaluation of x op y, there is no redefinition of x or y

意味着，若p处某表达式可用，则能将其替换为最后一次运算的结果.

可用于检测全局通用表达式

  • Abstraction

    Data Flow Values: all the expressions in aprogram

  • Safe-approximation:

    Transfer Function: $OUT[B]=gen_B\cup(IN[B]-kill_B)$

    Control Flow: $IN[B]=\cap_{P\ a\ predecessor\ of\ B}OUT[P]$

    到达该点前的每条路径都必须计算过x op y，因此使用交集.

L2 Intermediate Representation

Posted on 2021-10-27 In Static Analysis

程序源码直接拿来给静态分析器做静态分析是不合适的，像编译一样也需要在程序的中间表示(IR)上进行分析，这样能让静态分析算法比较简洁、高效.

编译的基本流程
1.1词法分析

词法分析(Lexical Analysis)会去检查是否输入的源代码是若干合法单词的组合.词法分析器统称Scanner，运行时需要关键字的词典，以及用正则表达式描述的词法规则.

词法分析会为每个合法的单词生成Token.

1.2语法分析

语法分析(Syntax Analysis)会去检查这些Token的组合形式是否符合语法规则.语法分析器统称Parser，相应的描述语法规则的方式是上下文无关文法(Context-Free Grammer).Context-Free Grammer的表达能力比Context-Sensitive Grammer弱，但是对几乎所有的编程语言而言，这样的表达能力就足够了，如果去使用后者反倒会更慢.

语法分析会生成抽象语法树(AST).

1.3语义分析

语义分析(Semantic Analysis)会去检查语义是否合理.和自然语言的语义不同，对编译器而言，只具备检查简单的语义的功能，如类型检查器(Type Checker)，相应的进行类型检查时描述规则的方法是Attribute Grammer.

语义分析会生成Decorated AST.

1.4转换

如果编译器有编译优化的功能，那么就要通过转换器(Translator) 转换成中间表示(IR) 的形式，这里的中间表示IR通常指的都是三地址码形式(3-Address Form).

生成IR之前的部分是编译器前端，生成IR之后的部分是编译器后端.编译优化就是静态分析的一种应用，静态分析是在IR的基础上做的，所以是属于编译器后端.这也说明了，要做静态分析，就不得不走完编译器前端的部分才能拿到IR，接着再在IR的基础上做分析.

1.5生成

为了能执行，要将上一步的中间表示，通过代码生成器(Code Generator) 生成机器码(Machine Code).

AST与三地址码的比较
AST

较高层，贴合编程语言的语法结构

通常依赖于具体的编程语言

适合快速进行类型检查

缺少控制流信息

3-address

较底层，接近于机器码

通常与编程语言无关

简洁、格式统一

易看出控制流信息

通常作为静态分析的IR的通用格式

三地址码简述

每条指令最多包含三个地址，其中地址指: 变量、常量、静态分析器自动生成的临时变量

如: x = y op z if x goto L

静态单赋值(SSA)

一种经典的IR格式，与普通的三地址码的区别在于: 每一个定义都使用了新的名称

phi-function

若出现控制流回合等情况，可使用 $\phi$​ 函数进行聚合

根据程序流程选择对应值

优缺点
  • 将flow-sensitive的指令变为flow-insensitive，即: 对指令的顺序不再敏感，可以提高分析速度.同时，因为本身带有的flow信息，可以较好保持flow-sensitive的精度.
  • 可以更精确找到其定义处，因为每条都是新定义
  • 引入太多变量和phi-function
  • 转换到机器码时，有很多低效代码
控制流分析

通常指: 构建控制流图**(Control Flow Graph)**

CFG作为静态分析的基础结构，其中的结点(node)可以是独立的三地址码，也可以是一个基本块(Basic Block).

基本块

Basic Block 是连续三地址码的最大序列，并满足以下条件:

  • 只能从开头处进入
  • 只能从结尾处离开
构建Basic Blocks

确定程序P中的leaders

  • P的第一条指令
  • 条件跳转或无条件跳转的目标指令
  • 紧跟在条件跳转或无条件跳转指令后

一个BB从一个leader开始到下一个leader开始前

基于BBs构建CFG

Basic Block作为结点，block A 有一条边指向block B 当且仅当:

  • 有一个条件跳转或无条件跳转从A的结尾跳至B的开头
  • B紧跟在A之后，且A的结尾不是无条件跳转

通常添加Entry、Exit两个节点，分别指向P的第一条指令及继承于P的最后一条指令.

L1 Introduction

Posted on 2021-10-27 In Static Analysis

背景：过去十年中，语言的内核几乎没有大的改变，但程序越趋复杂

挑战：面对大规模的复杂程序，如何确保其可靠安全？

静态分析的作用：

  • 程序可靠性
    • 空指针
    • 内存泄漏
  • 程序安全性
    • 私人信息泄露
    • 注入攻击
  • 编译优化
    • 死代码删除
  • 程序理解
    • IDE

定义：Static analysis analyzes a program P to reason about its behaviors and determines whether it satisfies some properties before running P.

然而，莱斯定理指出我们并无法获得一个准确完美的答案。

Rice‘s Theorem:

Any non-trivial property of the behavior of program in a r.e. language is undecidable.

于是，退而求其次，寻求尽可能有用的方案：

通过Overapproximate或者Underapproximate以获得Sound或Complete的结果。但会分别出现False Positives与False Negatives。

静态分析就是在确保soundness的基础上，在分析精度和速度上做一个平衡。

两个词简要概括静态分析（针对大多数的静态分析，即may analysis）

  • Abstraction
  • Over-approximation
